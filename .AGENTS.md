# .AGENTS.md

AI agent instructions for the Mesh codebase.

## Quick Reference

- **Stack**: Next.js 16, React 19, TypeScript (strict), Tailwind CSS 4, Supabase, shadcn/ui
- **Package Manager**: pnpm (required, enforced)
- **Dev Server**: `pnpm dev`
- **Lint**: `pnpm lint`
- **Tests**: `pnpm test` (unit, watch), `pnpm test:run` (unit, once), `pnpm test:e2e` (E2E)
- **Type Check**: `pnpm tsc --noEmit`
- **Build**: `pnpm build`
- **Add UI Components**: `pnpm dlx shadcn@latest add [component]`

## Conventions

- Follow conventional commit format: `type(scope): description` (e.g. `feat(matching): add location filter`)
- Pre-commit hook runs Prettier + ESLint on staged files via lint-staged
- Branch naming: `feat/`, `fix/`, `chore/`, `refactor/`, `docs/`, `test/` prefixes
- See `CONTRIBUTING.md` for full team conventions
- If the work contains planning, update specs in `spec/`

## CSS & Styling

- Use `size-*` shorthand when width = height (`size-4` not `h-4 w-4`)
- No `px` for font sizes — use `rem` or Tailwind classes (`text-[0.625rem]` not `text-[10px]`)
- No global `*` transitions — apply `transition-colors` to specific themed elements
- Custom styles go in `@layer` blocks to respect Tailwind specificity
- Semantic color tokens (`text-muted-foreground`) over raw Tailwind colors (`text-gray-500`)
- `focus-visible` over `focus` for keyboard-only focus rings
- Mobile-first responsive: base styles for mobile, enhance with `sm:`, `md:`, `lg:`

## Accessibility

- Never disable pinch-to-zoom (no `maximumScale: 1`)
- Interactive elements must be `<button>` or `<a>`, not clickable `<div>`s
- Dropdowns need `aria-expanded`, `aria-haspopup`, keyboard support (Escape, Tab, Enter/arrows)
- Icon-only buttons need `aria-label` or `sr-only` text
- No hover-only interactions — must have click/keyboard equivalent
- Color must not be sole conveyor of meaning — pair with text
- Minimum 44×44px touch targets on mobile
- Use Radix/shadcn primitives for complex interactive patterns
- **All interactive UI must be fully keyboard-navigable** — see the `keyboard-nav` skill for patterns

## Labels & i18n

All user-facing text strings must be centralized in `src/lib/labels.ts`. This is the single source of truth for UI copy and will be migrated to i18n message files when adding internationalization.

- Import `{ labels }` from `@/lib/labels` and reference strings via their key paths (e.g. `labels.landing.heroTitle`)
- Never use inline string literals for user-facing text in `.tsx` files
- Organize labels by feature area: `landing`, `meta`, `postingCreation`, `extraction`, `postingForm`, `github`, `onboarding`, `inputModeToggle`, `joinRequest`, `notification`
- Use functions for strings with dynamic content: `(n: number) => \`${n} items\``
- Use `as const` on enum-like sub-objects (e.g. `categoryOptions`, `experienceLevelOptions`) for type-safe lookups
- Feature labels like "AI Extract" are product names, not marketing copy — they stay as labels in this file

## Roadmap & Versioning

- When completing a feature listed in `spec/roadmap.md`, mark it as done (`- [x]`) and move it to "Implemented"
- When a milestone's features are all complete, bump `package.json` version and note in roadmap
- Feature PRs that implement roadmap items must update `spec/roadmap.md`

## Git Workflow

> **MANDATORY**: For any code changes, create a **git worktree** branched off `dev` (unless in the web environment). Never commit code directly to `dev` or `main`. Documentation-only changes on `dev` are acceptable.

- Never work directly on `main`, almost never directly on `dev` — always use a feature/fix/… branch
  - `dev` is fine for updating specs and documentation only
- If your work doesn't match the current branch's theme, create a new feature branch
- **Worktree setup**: `git worktree add ../Mesh-<branch-name> -b <branch-name> dev`, symlink `.env`, run `pnpm install`
- **One branch per task session**: When a plan has multiple sequential steps, use one worktree branch with separate commits per step. Only create additional branches for truly independent/parallel work that will become separate PRs.
- Commit and push freely when on your own feature/fix branch
- Only open a PR (target: `dev`) when work is complete and tests pass
- After completing work on an issue, update its status via `gh` CLI (e.g. `gh issue close N --comment "..."`)
- When creating related work, open new issues with `gh issue create`

## Worktrees

Two worktree locations are used:

- **Sibling worktrees** (`../Mesh-<branch-name>`) — for feature branches and main-session work. These are visible in the file manager and IDE, and the `/land` skill expects this layout.
- **`.claude/worktrees/`** — only for subagent `isolation: "worktree"` sessions. These are throwaway and auto-cleaned.

Setup notes:

- `.env` is not present in new worktrees — symlink from main repo: `ln -s /home/ajb/repos/Mesh/.env .env`
- Never `cp` the `.env` — symlinks stay in sync and avoid permission prompts
- `node_modules` is not present in new worktrees — run `pnpm install` after setup (fast due to pnpm's hardlinks). Always use `pnpm <cmd>` to run binaries, never `./node_modules/.bin/<cmd>` directly.
- Other gitignored files (e.g. local config, secret keys not in `.env`) must be **copied** into each worktree that needs them — they are not inherited and symlinks are only appropriate for files that must stay in sync
- Shell cwd may reset to the main repo after commands — use absolute paths or verify cwd

### Consolidating subagent branches

When a session spawns multiple subagents with `isolation: "worktree"`, each gets its own throwaway branch under `.claude/worktrees/`. If the work is part of a single planned effort:

1. Merge all subagent branches into the main session's worktree branch (respect dependency order — base branches first, then dependents)
2. Verify with `pnpm tsc --noEmit` + `pnpm test:run`
3. Land the single consolidated branch via `/land`
4. Clean up source branches after landing

Don't land subagent branches individually when they belong to a coordinated effort.

## Build Requirements

### Environment

Ensure `.env` (or `.env.local`) exists before running tests or dev server.

### Pre-Deployment Testing

Always run `pnpm build` before deploying to catch build-time errors.

### Stale Cache

"Element type is undefined" errors in dev mode can be caused by a stale `.next` cache, not a code bug. Fix: `rm -rf .next`, clear browser site data, restart `pnpm dev`.

### Suspense Boundaries

- `useSearchParams()` must be wrapped in `<Suspense>` to avoid build errors
- Extract component using `useSearchParams()` into separate component, wrap default export with Suspense

## Bug-Fixing Methodology

A bug is a signal — use it as a lens to examine the surrounding code and tests.

1. **Reproduce with a failing test**: Write or extend tests that fail due to the bug before fixing any code. Prefer general tests over narrow ones — test the _class_ of failure, not just the specific input that triggered it.
2. **Review surrounding tests**: While you're in the module, look at the existing tests. Are they shallow, missing edge cases, or testing implementation details instead of behavior? If you can elegantly generalize or strengthen them with low-to-medium effort, do it in the same PR.
3. **Fix the bug**: Make the minimal code change, confirm new and existing tests pass.
4. **Assess root cause**: Ask — _why was this bug possible?_ If it points to a fragile pattern, missing type constraint, or architectural issue that could cause similar bugs elsewhere:
   - If it's low-to-medium effort, fix it in the same PR
   - If it's larger, file an issue: `gh issue create --title "refactor: [describe the structural concern]" --label "tech-debt" --body "..."`
5. **Always tell the developer**: Surface what you found — the bug fix, any test improvements, and any architectural concerns — in the PR description or commit messages. Don't let the insight evaporate.

## Post-Task Testing Checklist

Before opening a PR, complete this checklist:

### 1. Run tests

- Run `pnpm test:run` — all unit tests must pass
- Run E2E tests selectively: consult `tests/e2e-map.md` to pick relevant specs (`pnpm test:e2e -- tests/e2e/<file>`). Run the full suite (`pnpm test:e2e`) before PRs or after schema/RLS/auth changes. Skip E2E for docs-only or pure-logic changes.
- Run `pnpm lint` and `pnpm tsc --noEmit` — no new errors
- If you changed UI components or pages, note what needs browser testing

### 2. Assess test coverage for your changes

Every test must justify itself: **what category of bug would it catch that TypeScript + lint wouldn't?** Don't test data passthrough, framework behavior (SWR, React rendering), or type-level contracts.

- **Non-trivial logic** (algorithms, state machines, security boundaries, data transformations): Must have tests targeting failure modes and properties, not happy-path examples
- **Data-passthrough code** (hooks that fetch and return, thin API wrappers): Don't unit-test — rely on e2e and integration coverage
- **Bug fixes**: Must include a regression test for the _class_ of bug, not just the specific case
- **Use `/write-tests`** for test design — it enforces adversarial test thinking and recommends using a subagent so tests don't mirror the implementation

### 3. Declare test gaps in your PR

- Fill in the **Test Gaps** section of the PR template honestly
- If any code paths remain untested, create a GitHub issue:
  ```bash
  gh issue create --title "test: [module/component name] lacks unit tests" \
    --label "untested" \
    --body "## What needs tests\n\n[describe the untested code]\n\n## Why not tested now\n\n[reason]\n\n## Suggested approach\n\n[how to test it]"
  ```

### 4. Flag browser-testing needs

- If your change affects UI rendering, interactivity, or responsive layout:
  - Add the `needs-browser-test` label to the PR
  - Fill in the **Browser Testing Needed** section of the PR template
  - List specific pages/flows to verify (e.g., "posting detail page at /postings/[id]")

### Test file conventions

- Unit tests: `src/<path>/__tests__/<module>.test.ts` (co-located)
- E2E tests: `tests/e2e/<feature>.spec.ts`
- Test utilities/factories: `tests/utils/` and `tests/fixtures/`
- Good test examples: `src/lib/matching/__tests__/scoring.test.ts` (algorithmic logic with bug-regression tests), `src/lib/calendar/__tests__/overlap-to-slots.test.ts` (date boundary logic), `src/lib/api/__tests__/guards.test.ts` (security boundaries)
- See the `write-tests` skill for test design methodology

## Supabase CLI

```bash
# Link to project (one-time setup)
supabase link --project-ref YOUR_PROJECT_REF

# Create migration
supabase migration new migration_name

# Apply migrations to remote database
supabase db push

# Check migration status
supabase migration list

# Pull remote schema changes
supabase db pull
```

When updating database model with semantic changes (new JSONB structures, dual-purpose fields, non-obvious constraints), update `docs/data-model.md`. Plain column additions don't need a doc update — the schema speaks for itself.

**Migrations must be applied when landing.** The `/land` skill runs `supabase db push` automatically after merging to `dev`. If landing manually, always run `supabase db push` after merging a branch that includes new migrations — landing code without applying its migrations leaves the database out of sync.

### Expand-Contract Migration Pattern

Use expand-contract for **breaking schema changes** (drop column, rename column, change type). This avoids downtime by splitting the change into two PRs:

1. **Expand** (PR 1): Add new column / stop reading old column in code. Deploy to `main`.
2. **Contract** (PR 2): Drop old column / index via migration. Deploy after Expand is live.

**Examples:**

- **Drop column `foo`**: PR 1 removes all code refs to `foo`. PR 2 migration: `ALTER TABLE t DROP COLUMN IF EXISTS foo;`
- **Rename column `old` → `new`**: PR 1 adds `new`, code reads `new` with `COALESCE(new, old)` fallback. PR 2 migration: backfill `new` from `old`, drop `old`.

**Rules:**

- Never drop a column that code still references
- Always deploy Expand first and verify in production before Contract
- Apply migrations to dev project first, verify, then prod

### pgTAP (RLS tests)

pgTAP tests live in `supabase/tests/`. Run against the **remote dev DB** — no Docker or `supabase start` needed:

```bash
source .env
PW=$(python3 -c "import urllib.parse; print(urllib.parse.quote('$SUPABASE_DB_PASSWORD', safe=''))")
supabase test db --db-url "postgresql://postgres:${PW}@db.wcfpmyiaauclgugjrntu.supabase.co:5432/postgres"
```

Gotchas:

- `SUPABASE_DB_PASSWORD` contains special chars — must be URL-encoded (snippet above handles this)
- `supabase test db --linked` is broken (pg_prove container can't reach remote). Always use `--db-url`
- `supabase test db` (no flags) works but requires `supabase start` (slow Docker startup)
- All test files need `SET search_path TO extensions, public, auth;` after `BEGIN;` for remote compatibility
- Tests with hardcoded row counts (e.g. `01_profiles_rls.sql`) will fail on the remote dev DB which has real data — only RLS policy logic tests are meaningful against remote

## Handoffs

The `.handoffs/` directory contains self-contained task briefs for delegating work to parallel Claude Code sessions (e.g. Claude Code on the web). Each file includes:

- Setup instructions (clone, branch, install)
- Task scope and sub-task breakdown
- Relevant database schema and existing code context
- Code patterns to follow (API routes, SWR hooks)
- Testing and verification checklist
- Commit conventions

To use: copy-paste the contents of a handoff file into a new Claude Code session. The agent will clone the repo and work independently on its feature branch. You'll need to provide `.env` separately since it's not in git.
